# M3.5 — Frontend Foundation — Design

Date: 2026-07-24

The first real UI. Today the frontend is one health page; this milestone lands the design
language, the component tiers, the data layer, the auth spine, and the first screen an
employee actually uses every morning.

It is scoped deliberately tight. The patterns established here — how a token becomes a
class, how a query key is built, how the session gates navigation — are what every later
screen copies, so they get built once, carefully, against a small surface.

## Where this fits

M0 skeleton, M1 pay primitives, M2 schema/auth/RBAC, M3 timekeeping ingestion, and M3.6
attendance adjustments are complete and merged. The backend this consumes is live: M2's
`/login`, `/me`, `/logout`, and M3's `/attendance/punch` and `/me/attendance`.

The foundation spec (`2026-07-23-hris-foundation-design.md`, §7) already settled the
architecture — one Next app, the design-language pipeline, three component tiers, the
five-file data layer, and routes mapped 1:1 onto the four backend scopes. This spec does
not relitigate that. It records what M3.5 specifically builds, and the decisions the
foundation spec left open.

## Decisions settled in brainstorming

| Decision | Choice | Why |
| --- | --- | --- |
| Scope | Tight foundation — design system, data layer, auth, and one screen | The patterns are the deliverable; more screens multiply review surface without adding foundational value |
| Product framing | A SaaS product, not an internal back-office tool | Sets the quality bar for the login, the shell, and the copy |
| Token source | `--radius: 0px` IBM/Carbon per `DESIGN.md` | The brief pins the visual direction; `DESIGN.md` is the authority |
| Token provenance | Port POS's `DESIGN.md` + hand-write `carbon.css` | Reproducible and sibling-consistent; no dependency on a generator that may have drifted |
| UI structure | Designed for this product, **not** ported from POS | POS's component structure serves a POS; only the token language is shared |
| App chrome | Carbon UI Shell — charcoal header + left scope-grouped side-nav | The authentic Carbon pattern; scales to the four scopes as they land, stays calm for an employee who only has "Me" |
| Login | Split canvas — charcoal brand panel + white form | The branded first impression a SaaS earns; a centered card reads as internal tooling |
| Token storage | `localStorage` | Survives reload for a tool kept open all day; first-party same-origin app with no third-party scripts |
| Attendance view | Month calendar grid | Reused by M4's holiday and schedule screens; matches how people think about a month of work |
| Punch placement | The hero of the attendance page, not its own route | The daily action and the record it produces belong together |

## 1. Architecture and the auth spine

**One client-rendered Next app.** `(auth)/login` is public; everything under `(app)/`
requires a session. Because auth is a `localStorage` bearer token rather than an SSR
cookie, the authed screens render client-side: a `QueryClientProvider` and a
`SessionProvider` sit at the `(app)` boundary.

**The token store** — `lib/session.ts` wraps `localStorage` (`getToken`, `setToken`,
`clearToken`) and exposes a subscription so a 401 can broadcast "logged out." No component
touches the token directly.

**`lib/api.ts` gains two behaviours**, and nothing else changes: it attaches
`Authorization: Bearer <token>` when a token exists, and on any `401` it clears the token
and broadcasts logout. The envelope handling built in M0 is already correct and stays.

**`useSession()` is the only source of role and scope truth.** With a token present,
`SessionProvider` fetches `GET /me` once — the `SessionResource`: `user`, `employee`
(nullable), `is_system_admin`, `has_reports`, `hr_offices`, `permissions` — and caches it
under a stable key. Navigation and route guards read that context. There is never a second
`/me` request. A `<Can permission=…>` wrapper is deliberately **not** built here — M3.5 has
no permission-gated control, only scope-gated navigation, and the component earns its place
when the first screen actually needs it.

**Client guards are UX; the server is the authority.** A guard in the `(app)` layout
redirects to `/login` when there is no token or `/me` refuses. Hiding a nav entry is
convenience — the policy already refused the request. Per-scope segments each gate on the
matching session flag when their screens land; in M3.5 only `Me` is populated.

**Login flow.** `POST /login` returns `{ token, user }`; the app stores the token, seeds
the session cache, and redirects to `/me/attendance`. `POST /logout` revokes the token
server-side and clears the store.

## 2. Design system and components

**Tokens.** `DESIGN.md` is copied to the repository root as the authority.
`src/styles/carbon.css` is hand-written from its front-matter and is the one place tokens
enter code: the palette (`#0f62fe` as the single accent, `#161616` ink, white canvas,
`#e0e0e0` hairlines), the IBM Plex Sans type scale, the spacing steps, and `--radius: 0px`.
Tailwind v4 consumes the custom properties.

**IBM Plex Sans is self-hosted via `next/font`,** not fetched from a CDN. No third-party
request at runtime, no layout shift, and the build stays deterministic offline.

**Three tiers, sized to what the screens consume.** POS's rule holds, with its inverse:
*if two screens render the same visual pattern it is a component; if only one screen
renders it, it stays in that page's file.*

- **Tier 1 — `components/ui/`**: `Button` (Carbon variants; label left, icon right),
  `TextInput` (filled, bottom rule, label and error), `InlineNotification`, `Skeleton`.
- **Tier 2 — chrome and generics**: `AppShell` (header + side-nav), `SideNav`
  (scope-grouped, driven by `useSession`), `SectionHeader`, `StatTile`, `Tag`,
  `EmptyState`.
- **Tier 3 — domain**: `MonthCalendar` and `DayCell`, and `Duration` — the one place
  integer minutes become human text, the visual mirror of `lib/duration.ts`.

Deferred until a screen needs them, each waiting on the backend that gives it data:
`DataTable`, `ConfirmDialog`, `Wizard`, `DayTypeTag`, `RequestCard`, `EmployeePicker`.

**The signature: a day cell is a ledger, not a summary.** `DayCell` renders the actual
in and out punch times rather than only a rolled-up total, because this system's thesis is
that the raw log is the record you show an inspector. A flagged punch keeps a visible
marker — verification is metadata on the row, never a gate at the door. Totals are set in
Plex Light 300 at display size, which `DESIGN.md` names as the brand signature.

**Copy is design material.** Sentence case, active voice, plain verbs; a control names
what happens when it is used and keeps that name through the flow. Failure states say what
went wrong and what to do, in the interface's voice, and never apologise. One copy rule is
load-bearing rather than stylistic: a failed sign-in reads **"That email and password
don't match"** for both a wrong password and an unknown email, because M2 deliberately
made those two cases indistinguishable — down to constant-time hashing — and the UI must
not leak what the API refuses to.

## 3. The data layer

Five files, no framework over React Query:

- `lib/api.ts` — the envelope client, extended with bearer auth and 401 handling.
- `lib/session.ts` — the token store.
- `lib/keys.ts` — the query-key factory.
- `lib/date.ts` — `YYYY-MM-DD` strings and month arithmetic. Calendar dates never become
  `Date` objects; a punch at 00:30 Asia/Manila belongs to the 30th, and a browser in
  another zone must not be able to disagree.
- `lib/duration.ts` — already in the repository from M1.

Thin hooks, one file per domain: `useSession`, `useMyAttendance(month)`, `usePunch()`.

Five conventions:

1. **Keys come from the factory, never a literal.** Invalidation is a typed prefix, not a
   guessed string.
2. **Hooks are thin** — a `useQuery` with a key and a fetcher, readable in ten seconds. No
   wrapper abstraction over React Query.
3. **Invalidate, don't write optimistically.** Optimistic updates are reserved for the
   approval queue when it lands; here a punch invalidates the month and the server's answer
   is the one rendered.
4. **`useSession()` is the only source of role and scope truth.**
5. **The server is the authority; client guards are UX.**

**The punch mutation generates a `crypto.randomUUID()` `Idempotency-Key` per attempt and
reuses that same key on retry.** The M3 endpoint requires the header, and reusing the key
is exactly what makes a double-tap or a flaky-connection retry safe — the second request
replays the first response instead of writing a second punch.

## 4. The screens

Three surfaces, not four:

**`(auth)/login`** — split canvas. A charcoal brand panel (wordmark in Plex Light 300, one
blue rule, a single line of positioning copy) beside a white form: work email, password,
and Carbon's signature button with the label left and the icon right. Inline notification
on failure.

**`(app)/me/attendance`** — the punch hero above the month ledger. The hero shows the
office-local clock, the current state (clocked in since 08:02, or clocked out), today's
total, and one primary action that is either "Clock in" or "Clock out." Below it, the
`MonthCalendar`: each day a `DayCell` carrying its punches, its total, and a flag marker
where verification failed. Month navigation is `YYYY-MM` in the URL, so a month is
linkable and refresh-stable.

**The shell** — `AppShell` with the charcoal header (product name, office context, user
menu with sign out) and the scope-grouped `SideNav`. Only **Me** is populated in M3.5;
`Team`, `Office`, and `Admin` appear as their screens land, each gated on the matching
session flag.

## 5. Testing and done-when

Vitest and Testing Library, the existing `api.test.ts` pattern (fetch mocked at the API
boundary). No Playwright — an E2E harness is its own decision and the backend's `scripts/e2e-*.sh`
already cover the API paths end to end.

- **`api.ts`**: attaches the bearer token when present, omits it when absent, and on a 401
  clears the token and broadcasts logout.
- **`lib/keys.ts`**: keys are stable and prefix-invalidatable.
- **`lib/date.ts`**: month arithmetic and `YYYY-MM-DD` handling never round-trip through a
  `Date` in a way that can shift a day.
- **`useSession`**: fetches once, exposes the scope flags, and drives which nav entries render.
- **Login**: a successful sign-in stores the token and redirects; a failed one renders the
  non-enumerating message and stores nothing.
- **Punch**: the mutation sends an `Idempotency-Key`, reuses it on retry, and invalidates
  the month on success.
- **Calendar**: punches render on their office-local date, a flagged punch shows its
  marker, and a month with no punches renders the empty state rather than an empty grid.
- **Quality floor, not announced but built**: responsive to mobile, visible keyboard focus,
  `prefers-reduced-motion` respected.

**Done when:** a seeded employee signs in at `/login`, lands on `/me/attendance`, clocks in
and sees the hero flip to "clocked in" with a live total, clocks out, sees the punch appear
on today's cell in the month ledger with its real in and out times, navigates to the
previous month, and signs out — with the whole surface rendered in the Carbon language from
`carbon.css` and no component reading a raw token or a literal query key.

## Non-goals for M3.5

The attendance-adjustment UI (M3.6's backend is merged; its screens ride with the requests
frontend pass). The employee roster and any `Team`/`Office`/`Admin` screen. Leave,
payslips, schedules, holidays, and cutoffs — their backends do not exist yet. A Playwright
E2E harness. Multi-tenancy, a mobile application, and offline punch capture, which are
non-goals for v1 generally.

## Open question

**The product name.** The mockups used "KronosHR" as a placeholder. The shell header, the
login brand panel, and the document title need a real name, or a decision to render
`config('hris.organization_name')` — the single-company framing makes the latter
defensible. Not blocking: the components take the name as a prop, so it is one constant to
change. Settle it before the login task is built.
